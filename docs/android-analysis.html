Android プロジェクト解析

Juml が Android Gradle プロジェクトをどう走査し、何を取り出して図にするか。入口 → スキャナ → 2 系統のパイプライン → データモデル → 各パーサ → 図生成 → 拡張ポイントまでソースから棚卸し。このアプリを改造・更新するための地図として読めるように構成しています。

入口 CLI / GUI / API 解析系統 2 本 Android パーサ 5 Android 系の図 6+
このページは src/main/java/juml/core/formats/android/・core/formats/java/AndroidProjectScanner.java・core/formats/uml/UmlGenerator.java・app/cli/AndroidCommands.java を直接棚卸ししたAndroid 解析の内部解説です。Java の字句・構文解析そのものの詳細は ソース解析パイプライン、CLI/GUI の使い方は 開発者リファレンス / GUI マニュアル を参照してください。

まず押さえる: Android 解析は「2 系統」ある

Juml が Android プロジェクトを読むとき、目的の違う 2 本のパイプラインが並走します。改造するときは「自分がどちらを触っているか」を最初に見極めるのが近道です。

系統入口読むもの作る中心モデル主な用途
① 構造解析
（Java/Kotlin/AIDL）		 UmlGenerator.extractFromProjectDetailed() .java / .kt / .aidl ソース List<JavaClassInfo> + ClassIndex クラス図 / シーケンス図 / コールグラフ / 継承図 など「コードの図」
② Android 資産解析 AndroidProjectAnalyzer.analyze() build.gradle(.kts) / settings.gradle / libs.versions.toml / AndroidManifest.xml / res/layout/*.xml / res/navigation/*.xml AndroidProjectAnalysis コンポーネント図 / Manifest 図 / 依存グラフ / Deep Link 図 / Navigation 図 / Layout 図

2 系統は独立して動かせますが、1 か所で交差します。①の extractFromProjectDetailed() は、最後に②を内部呼び出しして 2 つの情報を①側へ流し込みます。

  • Manifest マージ: ②で解析した <activity>/<service> 等の FQN を、①の JavaClassInfo.androidComponentType に書き戻す（→ クラス図でコンポーネント種別を色分けできる）。
  • 依存インデックス: ②の Gradle 依存宣言から DependencyJarIndex を組み、①のシーケンス図などで外部ライブラリのクラスを遅延ロードできるようにする。
覚え方: 「コードの形」を描きたいなら系統①、「アプリの構成（マニフェスト・ビルド・画面定義）」を描きたいなら系統②。両方が要る図（クラス図でコンポーネント色分け、シーケンス図で外部解決）は①が②を呼ぶことで成立します。

入口 — CLI / GUI / API のどこから解析が始まるか

CLI（app/cli/AndroidCommands）

CliDispatcher がオプションを見て AndroidCommands の各ハンドラへ振り分けます。Android 関連の主なフラグ:

フラグ（short / long）ハンドラ系統出力
-g / --gradlehandleGradleInput②1 ファイルの Gradle 解析テキスト
-m / --manifesthandleManifestInput②1 ファイルの Manifest 解析テキスト
-d / --component-diagramhandleComponentDiagram②コンポーネント図
-M / --manifest-diagramhandleManifestDiagram②Manifest 図
-D / --deeplink-diagramhandleDeepLinkDiagram②Deep Link 図
-G / --dependency-graphhandleDependencyGraph②Gradle 依存グラフ
--summaryhandleSummary②プロジェクト概要 Markdown
--settingshandleSettings②派生設定（Preferences/SharedPreferences）レポート
--action-maphandleActionMap②派生UI アクションマップ

--all は Android プロジェクトの「ひと通り」を 1 コマンドで出すフラッグシップ経路で、handleAll() が次の順に生成します: [1] summary.md[2] component-diagram.svg[3] manifest-diagram.svg[4] deeplink-diagram.svg[5] dependency-graph.svg[6] class-diagram.svg（①の Java/AIDL 走査）[7] methods.txt[8] sequence-diagrams/（ライフサイクル起点）。系統①と②が両方走る唯一の CLI 経路です。

GUI（app/uml/DiagramService → DiagramKind）

GUI では各タブが DiagramKind を 1 つ持ち、DiagramService が種別に応じて系統①/②を呼び分けます。Android 系の DiagramKind:

COMPONENTMANIFESTDEPENDENCYNAVIGATIONLAYOUTSCREEN_FLOWSOONG（Android.bp） ＝ 系統② / 派生。CLASSSEQUENCECALLGRAPHINHERITANCEPACKAGECOMMONACTIVITYMODULE ＝ 系統①。

プロジェクトを開いた瞬間の重い解析（系統①の並列パース）は ProjectLoader が SwingWorker で背景実行し、結果は ProjectAnalysisCache にキャッシュされます（流れは ソース解析・Stage A/B の図5を参照）。②の Android 資産解析は比較的軽いので、図を開くタイミングで都度 AndroidProjectAnalyzer.analyze() が走ることが多いです。

API（コードから直接）

// 系統① 構造解析（並列・進捗・キャンセル対応）
UmlGenerator.ProjectParseResult r =
    UmlGenerator.extractFromProjectDetailed(
        root, scanOpts, errListener, progress, cancel,
        /*mergeManifest=*/true, UmlGenerator.ParseMode.FULL);
List<JavaClassInfo> classes = r.getClasses();

// 系統② Android 資産解析
AndroidProjectAnalysis a = AndroidProjectAnalyzer.analyze(root, errListener);
a.getGradleByModule();        // モジュール → Gradle 情報
a.allManifests();             // 全モジュールの Manifest
a.allComponents();            // 全コンポーネント

入力の収集 — AndroidProjectScanner

両系統の最初の一歩は AndroidProjectScanner.scan(root, opts) によるファイル列挙です。java.nio.file.Files.walkFileTree ベースで、AOSP 級の巨大ツリーでも詰まらないように設計されています。

何を拾うか（Options のフラグ）

accept() が拡張子・ファイル名・親ディレクトリ名で取捨選択します。.java は常に対象、それ以外は opt-in です。

Options フラグ既定拾うもの
（常時）—*.java
includeKotlinfalse*.kt / *.kts
includeAidlfalse*.aidl
includeHidlfalse*.hal（AOSP の HAL）
includeVintffalsemanifest.xml / compatibility_matrix.xml（VINTF。ルート要素で AndroidManifest と区別）
includeGradlefalse*.gradle / *.gradle.kts
includeManifestfalseAndroidManifest.xml（小文字一致）
includeLayoutfalse親が layout / layout-* の *.xml
includeNavigationfalse親が navigation / navigation-* の *.xml

系統②の AndroidProjectAnalyzer は includeGradle/Manifest/Layout/Navigation をすべて true にして 1 回だけ scan() し、戻り値のファイルリストを名前で振り分けます。系統①の extractFromProjectDetailed() は includeAidl=true / includeKotlin=true を強制してから .java/.kt/.aidl だけを並列パースに回します。

何を捨てるか（除外ディレクトリ）

  • DEFAULT_EXCLUDED_DIRS: build / .gradle / .idea / .git / out / bin / node_modules / .kotlin / captures / .cxx。preVisitDirectory で SKIP_SUBTREE。
  • AOSP_EXTRA_EXCLUDED_DIRS（useAospDefaults=true 時に合成）: prebuilts / out-soong / .repo / test_mapping / .cache。
  • テストソース: includeTests=false（既定）なら src/test / src/androidTest 直下を除外。

暴走防止のつまみ

maxFiles（上限到達で TERMINATE）maxDepth（再帰深さ、内部で +1 換算）cancelToken（preVisit/visit で都度チェック）followSymlinks（既定 false）。権限拒否は visitFileFailed で握りつぶして走査継続。読み込みは readFile() がUTF-8 固定で行単位に読み、改行を \n に正規化します。

モジュール名の推定 — inferModuleName()

マルチモジュール構成を扱うため、各ファイルがどの Gradle モジュールに属すかを相対パスから推定します。これは系統①（ClassIndex のモジュールキー）と②（gradleByModule 等のキー）の両方で使う共通ロジックです（AndroidProjectAnalyzer.inferModuleName(root, file)、static public）。

ファイルパス（root 相対）モジュール名
app/build.gradleapp
core/common/build.gradlecore:common
build.gradle（ルート直下）:root
app/src/main/AndroidManifest.xmlapp（src/... 以下を除去）

要点: ファイル名を除いた親パスを取り、/src/ 以降（ソースセット部分）を切り落とし、残りの / を : に変換します。空になればルートを表す :root。settings.gradle は特別扱いで、analysis.setRootSettings() に :root として格納されます。

ソースセット / configuration qualifier の推定

Android は同じ論理ファイルを flavor / build type / 画面構成ごとに複数持てます。AndroidProjectAnalyzer はパス階層からこれを推定し、各 info に付与します。

ヘルパ入力例結果
inferSourceSet（Manifest）src/debug/AndroidManifest.xmldebug（src/<name>/ の name、なければ main）
inferLayoutSourceSetsrc/main/res/layout/a.xmlmain
inferLayoutConfigQualifierres/layout-sw600dp-v21/a.xmlsw600dp-v21（素の layout/ は ""）

コンポーネント図では main 以外のソースセットを <<src:debug>> のように付記し、Layout 図では qualifier 違いを別ノードとして扱えます。

系統① 構造解析 — 並列パースの流れ

中心は UmlGenerator.extractFromProjectDetailed(root, opts, listener, progress, cancel, mergeManifest, mode)。Android プロジェクトの巨大なソースを現実的な時間で読むための心臓部です。

  1. スキャン: includeAidl/Kotlin=true で AndroidProjectScanner.scan()。進捗 "Scanning files..."。
  2. ソースルート推定: FULL モード時のみ、.../src/<set>/(java|kotlin)/ を正規表現 (deriveSourceRoots()) で抽出し、JpSolver.fromSourceRoots() でシンボル解決器を構築（呼び出し先 FQN/シグネチャ解決用）。HEADERS_ONLY では solver=null。
  3. 並列実行: ワーカ数 = max(2, CPU-1) の daemon スレッドプール。ExecutorCompletionService に .java/.kt/.aidl 1 ファイル = 1 ParseTask を投入。
  4. 個別パース: ParseTask.call() が readFile() → extractFromSource()（FULL）または extractHeadersFromSource()（HEADERS_ONLY）。拡張子で AidlParser / KotlinLightScanner / JavaStructureExtractor に分岐（詳細は ソース解析）。各クラスに AaosPattern / JetpackPattern のステレオタイプを付与。
  5. 集約: 完了順に結果を all リストへ、ClassIndex.put(c, file, module) でインデックス化。進捗を done/submitted で通知。キャンセル時は shutdownNow()。
  6. Manifest マージ（mergeManifest=true 時）→ 次節。
  7. 依存インデックス構築 → 後述。
  8. 返却: ProjectParseResult(classes, ClassIndex, DependencyJarIndex)。
注意: 並列実行のため 結果リストの順序は元のファイル順と一致しません。順序に依存するテスト・処理は自前で再ソートしてください（Javadoc にも明記）。JavaParser はスレッド非安全なので JpSolver はワーカ毎にパーサを作ります。解決失敗（UnsolvedSymbolException 等）は無視して best-effort で続行します。

Manifest マージ — ②の結果を①へ流し込む

mergeManifestInto(classes, root, listener) が系統②を内部で 1 回呼び（AndroidProjectAnalyzer.analyze()）、得た各 AndroidComponentInfo の FQN を classes の対応 JavaClassInfo に突き合わせて androidComponentType を設定します。

  • マッチ判定 matches(): まず FQN 完全一致、ダメなら 単純名一致（Manifest 側が package を省略した場合の救済）。
  • 設定値は component.getKind().label()（Activity / Service / BroadcastReceiver / ContentProvider）。
  • マージ後、ClassIndex の header 側にも同じ値を反映（all と header は別インスタンスのため個別に書き戻し）。

これにより、クラス図やパッケージ図で「このクラスは Activity」というステレオタイプ／色分けが可能になります。AndroidManifest.xml に現れない Fragment などは、別途 AndroidSuperclassDetector が ClassIndex の継承チェーンを辿って補完します（ステレオタイプ判定参照）。

依存 JAR/AAR インデックス — DependencyJarIndex

buildDependencyIndex() が系統②を呼んで全モジュールの GradleDependency をフラットに集め、DependencyJarIndex.build() に渡します。これは「外部ライブラリのクラスを必要になったときだけ読む」遅延カタログで、シーケンス図などで外部 participant を <<external>> として正しく扱うために使われます（クラス本体は ExternalClassReader が ASM で宣言ヘッダのみ抽出。外部 JAR ヘッダ抽出）。失敗時は空の DependencyJarIndex を返すので、利用側は null/空チェックして best-effort で続行します。

系統② Android 資産解析 — AndroidProjectAnalyzer.analyze()

Gradle / Manifest / layout / navigation を 1 つの AndroidProjectAnalysis に束ねるファサード。処理順は依存関係順に組まれています。

  1. 0. Version Catalog 先読み: gradle/libs.versions.toml があれば VersionCatalogParser で読み、後段の Gradle パースに catalog として渡す（libs.xxx を実値に解決するため先に読む）。
  2. 1. settings.gradle を先にパース: GradleScriptParser.parse() でサブプロジェクト構造を取得し :root として setRootSettings()。
  3. 2. 各ファイルを名前で振り分けてパース:
    • build.gradle(.kts) → GradleScriptParser → gradleByModule[module]
    • navigation/ 配下の *.xml → AndroidNavigationGraphParser → navigationsByModule
    • layout/ 配下の *.xml → AndroidLayoutParser → layoutsByModule（sourceSet + qualifier 付与）
    • androidmanifest.xml → AndroidManifestParser → manifestsByModule（sourceSet 付与）
    各パースは try/catch(RuntimeException) で囲い、1 ファイルの失敗は listener.onError に通知して残りを継続。
  4. 3. package のバックフィル backfillPackageFromGradle(): モダンな Manifest は package 属性を持たないことがあるため、同モジュールの gradle.namespace で補い、rewriteFqn() でコンポーネント名を完全 FQN に再解決（先頭 . や package 無しの単純名を namespace + name に展開）。
  5. 4. サマリーログ: gradle/manifest/layout/navigation の件数を listener に通知して AndroidProjectAnalysis を返す。
順序が重要: ① TOML → ② settings → 各 build.gradle、の順は「libs.* を解決するには catalog が先」「モジュール構造を知るには settings が先」という依存から来ています。ここを崩すと依存解決やモジュール対応が壊れるので、改造時は順序を保ってください。

各パーサの詳細

系統②が使う 5 つのパーサ。Android のビルド/リソースファイルは「厳密な文法より現場のパターン」が支配的なので、best-effort（壊れても止まらない）方針で書かれています。

パーサ方式抽出するもの / 特徴
GradleScriptParserコメント除去＋文字列リテラル保持＋ブロック追跡＋正規表現（best-effort）plugins / applicationId / namespace / 各種 SDK / versionCode・Name / buildTypes / productFlavors / signingConfigs / flavorDimensions / dependencies（scope・group・name・version・project(":x") モジュール参照）/ subprojects。libs.* 参照は VersionCatalog で実値解決。Groovy/KTS 両対応。
VersionCatalogParser軽量 TOML[versions] / [libraries] / [plugins]。単純値＋インラインテーブル対応。bundles・テーブル配列は未対応（拡張余地）。
AndroidManifestParserDOM（XXE 無効化: DOCTYPE 禁止・外部エンティティ無効）<application> の多数の属性、SDK、4 種コンポーネント（exported/enabled/launcher/intent-filter）、uses-permission / permission / uses-feature / meta-data。
AndroidLayoutParserDOM 再帰View 階層を LayoutViewNode ツリーに（viewType / id / text / width / height / isViewGroup / <include> / <fragment> / children）。
AndroidNavigationGraphParserDOMstartDestination、Destination（type・className・arguments・deepLinks）、Action（destination・argOverrides・navOptions）。
セキュリティ: XML 系パーサ（Manifest / Layout / Navigation / VINTF）は外部エンティティ展開（XXE）を無効化しています。新しい XML パーサを足すときは同じ防御設定（AndroidManifestParser の設定をコピー）を必ず入れてください。

AOSP / AAOS の踏み込んだ解析戦略は Claude Code のスキル（/aosp-juml-analyzer / /aaos-juml-analyzer）にもまとまっています（Claude Code 連携）。

改造 ①: 新しいファイル種を解析対象にする

例「*.pro（ProGuard）も拾いたい」。触る場所は AndroidProjectScanner だけです。

  1. Options に public boolean includeProguard = false; を追加。
  2. accept() に if (opts.includeProguard && name.endsWith(".pro")) return true; を追加。
  3. 必要なら除外ディレクトリ（DEFAULT_EXCLUDED_DIRS）も見直す。
  4. 呼び出し側（AndroidProjectAnalyzer.analyze() の opts 設定、または②の振り分け for ループ）でフラグを立て、ファイル名で分岐を足す。
opt-in（既定 false）にするのが鉄則です。.java 以外を常時 true にすると AOSP 級ツリーで走査コストが跳ね上がります。テストは AndroidProjectScannerTest（既存パターン: 「フラグ true で拾う / 既定では拾わない」の 2 ケース）に倣ってください。

改造 ②: 新しいパーサ／資産を足す

例「res/menu/*.xml を解析してメニュー図を出したい」。系統②の各層に対応物を作るのがパターンです。

  1. モデル: core/formats/android/AndroidMenuInfo.java（フィールド＋getter/setter、filePath/moduleName/sourceSet/key を既存 info に合わせる）。
  2. パーサ: AndroidMenuParser.parse(content, listener)。XML なら AndroidManifestParser のXXE 無効設定をコピー。失敗は throw せず listener 通知＋空返し。
  3. スキャナ対応: 改造①の手順で includeMenu と menu/ 判定を追加。
  4. 集約: AndroidProjectAnalysis に menusByModule Map と allMenus() を追加し、AndroidProjectAnalyzer.analyze() の振り分けループに分岐を 1 本足す（moduleName・sourceSet 付与を忘れず）。
  5. テスト: 単体（パーサ）＋ AndroidProjectScannerTest ＋ 必要なら RealWorldSampleParseTest。

改造 ③: 新しい図種を足す（CLI と GUI 両方）

CHANGE.md の「Soong 図を GUI に追加」が良い実例です。基本の足し場所:

  1. 生成器: PlantUmlMenuDiagram.generate(analysis, opts): String。出力は PlantUML 文字列で、レンダリングは既存の PlantUmlRenderer 等に任せる。
  2. CLI: CliOptions にフラグ（new Option(null, "menu-diagram", false)）、CliDispatcher に分岐、AndroidCommands.handleMenuDiagram() を追加。--all に載せるなら handleAll() のステップも増やす。
  3. GUI: DiagramKind に列挙子（ラベル）を追加 → DiagramService に generateMenuPuml() を実装し種別で呼び分け → ToolBarBuilder/MenuBarBuilder に項目 → TreeNodeIcon でアイコン割当 → 「Android 系」セット（DIAGRAMS_ANDROID）に追加。
  4. 空入力時の振る舞い: 対象が 0 件なら空図でなく案内ノート付き PlantUMLを返す（Soong 図の踏襲。ユーザが「壊れた」と誤解しないため）。
  5. テスト: DiagramServiceTest に「データあり → 期待ノード」「データ無し → 案内ノート」の 2 ケース。
GUI のアーキテクチャ方針（CLAUDE.md）に従い、図種切替は常にアクティブタブ起点で書きます。DiagramState はグローバル最小、タブ固有状態はタブ側へ。Home 前提の分岐は新規追加しないでください。

改造時の注意点（ハマりどころ）

  • 解析順序を崩さない: ②は「TOML → settings → build.gradle」の順。libs.* 解決とモジュール対応がこの順序に依存します。
  • best-effort を壊さない: 1 ファイルの失敗で全体を止めない。新パーサも try/catch(RuntimeException) ＋ listener.onError 通知で続行。
  • 並列性: 系統①はマルチスレッド。共有状態を足すなら ClassIndex のように concurrent 安全に。JavaParser はワーカ毎に生成（スレッド非安全）。
  • 結果順序は不定: 系統①の classes はファイル順でない。順序が要るなら自前ソート。
  • FQN の揺れ: モダン Manifest は package 無し。namespace バックフィル後の FQN を前提に。マッチは FQN→単純名のフォールバックを使う（matches() 参照）。
  • 巨大ツリー対策: AOSP 級では useAospDefaults / maxFiles / maxDepth / cancelToken を必ず尊重。新しい走査ロジックもこれらを貫通させる。
  • XXE 防御: XML パーサは外部エンティティ無効化を必ず入れる。
  • Stage A/B: ヘッダのみ（HEADERS_ONLY）では fields/methods/呼び出し列が空。本体が要る図は ClassIndex.detail() で昇格してから使う（参照）。
