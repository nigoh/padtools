アーキテクチャ

Juml は大きく CLI/GUI エントリ層・UML GUI 層・解析エンジン (core) の 3 層で構成されます。エントリは juml.Main。引数なし（または図モード非該当）で Swing GUI が起動し、図モードフラグがあれば CLI として単発／一括出力を行います。

解析パイプライン

中核モデルは JavaClassInfo（Kind: CLASS / INTERFACE / ENUM / ANNOTATION / AIDL_INTERFACE / RECORD / MODULE）。プロジェクト解析は次の流れで進みます。

  1. 列挙 — AndroidProjectScanner.scan が walkFileTree でソースを収集（キャンセル・上限対応）。
  2. 並列パース — UmlGenerator.extractFromProjectDetailed がスレッドプールで各ファイルを処理。ParseMode は FULL / HEADERS_ONLY。
  3. ディスパッチ — 拡張子で分岐：.aidl→AidlParser、.kt/.kts→KotlinLightScanner、その他→JavaStructureExtractor。
  4. 字句解析（前処理） — JavaLexer はコメント/文字列/数値を切り出す簡易レキサ。Java 本体は JavaParser に委譲し、Lexer は Unicode エスケープ展開と AIDL/HIDL のトークン供給に使用。
  5. 構造抽出 — JavaParserFrontend が JavaParser + SymbolSolver を BLEEDING_EDGE レベルで寛容にパース。構文エラーでも部分結果を返し、壊れたソースは正規表現で型ヘッダだけ復元。
  6. シンボル解決 — FULL 時のみ JpSolver を構築し、呼び出し先 FQN を Call.resolvedOwnerFqn に解決。
  7. メソッド本体 — StatementAdapter が if/while/for/do/switch/synchronized/try を Block/Branch の構造化ツリーへ。シーケンス図・アクティビティ図・メトリクスの源。
  8. 後処理 — AAOS カテゴリ付与、Jetpack ステレオタイプ付与、Manifest マージ。
  9. 生成 — PlantUml*Diagram.generate へ。
Stage A / B 二相: extractHeadersOnly でヘッダのみ軽量保持し、ClassIndex が FQN→ヘッダを全件メモリ保持、必要時に詳細（Stage B）へオンデマンド昇格。大規模プロジェクトの初期表示を高速化する。

AIDL / HIDL 対応: あり。AidlParser が in/out/inout・oneway を抽出し Kind.AIDL_INTERFACE を生成。HidlParser は .hal のバージョン付きパッケージと generates 句を扱う。

解析 → 図解化のディスパッチ

GUI のトグル / メニューは DiagramController.selectDiagramKind → DiagramService.generatePuml(request, cache) 経由で各生成器へ振り分けられます。入力源は 3 系統に分かれ、パースの起点（解析済みモデルの再利用か、生成時の再走査か）が異なります。

① 解析済み Java モデル

List<JavaClassInfo> を再利用。applyScope で絞り込み、promoteToDetailed（ClassIndex#detail）で必要分だけ Stage B 昇格してから描画。
Class / Inherit / Package / Common / Module / Sequence / Activity / Call Graph

② Android / Gradle 解析

AndroidProjectAnalysis（Manifest・layout・navigation・Gradle のパース済み結果）から生成。Layout / Navigation は findLayoutByKey / findNavigationByKey で対象 1 件を引く。
Component / Dependency / Manifest / Layout / Navigation

③ 生成時に再走査

キャッシュではなく cache.getProjectRoot() 配下を改めて走査。Screen Flow は IntentNavigationDetector（regex）、Soong は AndroidBpParser（軽量 Blueprint パーサ）。
Screen Flow / Soong

CLI リファレンス

エントリは juml.Main。args[0] == "index" のときのみ IndexCommand へ早期分岐し、それ以外は CliOptions.parse → CliDispatcher.dispatch。どのモードにも当たらなければ GUI が起動します（第 1 位置引数を初期プロジェクトとして渡す）。

# GUI 起動（引数なし、または初期プロジェクト指定）
java -jar Juml.jar [projectDir]

# クラス図を SVG 出力
java -jar Juml.jar -c -o class.svg ~/AndroidStudioProjects/MyApp

# メソッド起点のシーケンス図
java -jar Juml.jar -q "MainActivity.onCreate" -o seq.svg MainActivity.java

# 全成果物を一括出力（要 -o ディレクトリ）
java -jar Juml.jar --all -o ./out ~/AndroidStudioProjects/MyApp
OptionParser の制約: 短縮形の連結（-vc）や --opt=value 形式は 非対応（--opt value と空白区切りで指定）。例外として index サブコマンドの --db=path のみ手動対応。未知オプションは exit 1。

出力フォーマットの決定

  • PlantUML 系：-o 拡張子が .svg なら同梱 PlantUML で SVG レンダリング、それ以外は PlantUML テキスト。SVG 失敗時は同名 .puml をサイドカー保存して exit 2。
  • Markdown / テキスト系：そのまま書き込み（-g/-m/--summary/--action-map/--ref-find 等）。
  • 二重成果物系：拡張子で .md/.puml/.svg を分岐。拡張子なしなら .md と .puml を両方出力（--impact/--data-flow/--screen-flow/--vhal-flow/--android-bp）。

index サブコマンド（SQLite インデックス構築）

java -jar Juml.jar index <projectRoot> [--db <path>]

図生成系の CLI は 毎回ファイルシステムを新規走査し SQLite を読まないのに対し、index サブコマンドのみが ~/.juml/cache/<hash>/index.db を構築・増分更新します。2 回目以降は mtime/size 差分（IncrementalScanner）で stale ファイルのみ再パース。完了時に [juml] index complete: ... を stderr へ出力。

ツールバー

上段アクション：OpenSaveRefreshSearch。下段は DiagramKind.values() 全 15 種の排他トグル（Class / Package / Sequence / Activity / Component / Dependency / Manifest / Layout / Common / Navigation / Module / Inherit / Call Graph / Screen Flow / Soong）。

タブのライフサイクルとフォーカス連動

  • 重複抑止: openTabs(LinkedHashMap) に key が既存ならフォーカスのみ移動。新規は固定タブ 6 本の手前に挿入。METHOD タブは kind:Class.method、CLASS タブは CLASS:fqn で識別。
  • 連動フロー: タブ選択 → handleTabSelectionChanged → onTabFocused(FocusedTab) → syncToFocusedTab（ツリーをハイライト）+ setCurrentKind + reflectKindInToolbar（メニューラジオ + トグルを選択）。アクティブタブ↔サイドバー↔ツールバーが常に同期。
  • 描画: 各 DiagramTab が自前の SwingWorker で DiagramService.generatePuml → PlantUmlSvgRenderer.render。CardLayout で「Rendering… / プレビュー / 失敗・空」を切替。
  • 図種切替: メソッド系タブ（Sequence/Activity/CallGraph）にフォーカス中は、同じ Class.method の別図種を新タブで開く。
  • タブ操作: ×ボタン / 中クリック / 右クリック「Close / Close Others / Close All」。固定 6 タブは常に残る。

ユーティリティタブ（解析）

Impact

シンボル + Depth(1–10, 既定 3) → Layer 別ツリー（リスク・スコア・理由）。

References

シンボルの逆参照を JTable（Caller class/Method/Kind/File）。--ref-find 相当。

Func Diff

2 メソッドを LCS / Levenshtein / Jaccard で比較し Markdown レポート。

エンティティ検索: Class/Method/Field 横断、Kind フィルタ + 部分一致（150ms debounce）。OK=通常選択、Drill-down=DETAILED 詳細図。
スコープ: Include/Exclude パッケージ・モジュール・クラス名 regex・Max classes・Seed neighbor hops・関連種別・可視性・外部除外・Parse mode。
設定の自動保存: プロジェクトごとに style / sequence / classDiagram 設定を SQLite に保存し、開き直すと復元。ウィンドウ位置・サイズも保存。

専門解析パッケージ

impact

ReferenceIndex 上を BFS で逆方向（呼出元・継承元・型参照元）に辿る。影響度スコア 1/(layer+1)。

funcdiff

呼び出し列を LCS 類似度 / Levenshtein / Jaccard で比較。マッチ信頼度は receiver*0.4 + firstArg*0.3 + position*0.3。

dataflow

Room @Entity/@Dao/@Database をアノテーション正規表現で検出 → ER 図 / Markdown。

screen

Intent / Compose / NavGraph の 3 経路で画面遷移を検出し、入次数 0 起点から多段パス列挙。

AOSP（プラットフォーム）

AndroidBpParser（Soong）/ AndroidMkParser（legacy）/ SelinuxPolicyParser（.te/.cil）/ VintfManifestParser（HAL 宣言・カーネル要求）/ RroOverlayDetector（<overlay> の targetPackage/isStatic/priority）。

永続化・SQLite インデックス

配置は ~/.juml/cache/<shortHash>/index.db（Windows は %LOCALAPPDATA%/Juml/cache/...）。shortHash はプロジェクトルートの canonical path のみから計算し、ファイル差分は DB 内の files.mtime/size で検知します。

  • マイグレーション無し：SCHEMA_VERSION 不一致時は index.db.discarded-* へ退避して再構築。PRAGMA は WAL / synchronous=NORMAL / foreign_keys=ON。
  • 主要テーブル：modulesfilesclassesclass_interfacesclass_importsfieldsmethodsrefsunresolvedmanifestscomponentsintent_filtersaidl_interfacesaidl_methodsaidl_bindingsexternal_endpoints
  • 書き込み：IndexWriter が 1 ファイル = 1 トランザクション（path DELETE → CASCADE → INSERT → ClassIngestor 系）。集約系は毎回 wipe → 再構築で冪等。
  • 増分：IncrementalScanner が files と現 FS を mtime + size で突合（ハッシュ不使用）。GradleProjectScope がモジュール境界を拾い AOSP 級ツリーの全走査を回避。

PlantUML 描画

入口は PlantUmlRenderer.renderSvg。同梱 PlantUML jar の SourceStringReader.outputImage(FileFormat.SVG) を使用します。

  • Smetana: injectLayout が @startuml 直後に !pragma layout smetana を挿入し、Graphviz/dot 不在環境でも描画可能に。dot 検出時は挿入をスキップ（GraphvizLocator がプロパティ → 環境変数 → 同梱 → PATH の順で探索）。
  • エラー検出: PlantUML の "An error has occured" マーカーを先頭 8KB で検査し、検出時は例外を送出（壊れた SVG 保存を回避）。
  • 日本語フォント: スタイル未指定時 JapaneseFontSupport.defaultFontFamily() を skinparam defaultFontName で補い文字化け防止。
  • Batik: SVG → ラスタ表示 / GraphicsNode 描画は GUI 層（Apache Batik）。core はバイト列を返すのみ。
