GUI 操作マニュアル

Juml の Swing GUI（VS Code 風タブ中心モデル）の使い方。起動からエクスポート、ショートカットまで。

対象: GUI 図種 15 タブ = エディタ

起動と画面構成

起動方法

# GUI を起動
java -jar Juml.jar

# 起動と同時にプロジェクトを解析開始
java -jar Juml.jar ~/AndroidStudioProjects/MyApp

変更点 (2.x): ツールバー上段はアクション 4 種のみ。Scope / Clear Scope は Diagram メニュー、Zoom は View メニューから操作します。履歴を戻る Back ボタンは廃止されました。

プロジェクトを開く

Open Project ダイアログ

File → Open Project…（または Ctrl+O、ツールバー Open）で開きます。最近使ったプロジェクト一覧と「新しいフォルダを参照…」が表示されます。

操作動作
リストをダブルクリックそのプロジェクトをロード
リスト選択 → 開くそのプロジェクトをロード
リスト選択 → 一覧から削除（または Delete キー）最近開いたプロジェクト一覧から取り除く（確認あり）。ディスク上のフォルダ・ファイルは削除されません
新しいフォルダを参照…フォルダ選択ダイアログで選ぶ

File → Open Recent

最近開いたプロジェクトが最大 10 件表示されます。存在しないディレクトリは自動的に無効化（グレー表示）されます。

プロジェクトのロード

  1. フォルダ選択でバックグラウンド解析が開始（全面 GIF オーバーレイ表示）
  2. ステータスバーに Analyzing 1234 / 56789 のような進捗
  3. 完了後、左ツリーにモジュール・パッケージ・クラス・メソッドが展開
  4. 前回のプロジェクト設定（スタイル・プリセット）が自動復元され、既定で Package 図タブが 1 枚開く
ヒント: 解析途中で中断するには File → Cancel Loading を使います。

各トグルがどう解析して図解化するか

各トグルは DiagramService.generatePuml から対応する PlantUml*Diagram 生成器にディスパッチされます。入力源は大きく 3 系統です。

  • 解析済み Java モデル（JavaClassInfo 群）: ロード時に構築済みのモデルを再利用。スコープで絞り込み、必要なクラスだけ Stage B（詳細）へ昇格してから描画。
  • Android XML / Gradle 解析（AndroidProjectAnalysis）: AndroidManifest・layout・navigation・build.gradle を解析した結果から生成。
  • プロジェクト再走査: Screen Flow / Soong のみ、生成時にプロジェクトルート配下のソース（Java/Kotlin・Android.bp）を改めて走査して解析。
図種入力源解析方法図解化（PlantUML へのマッピング）
ClassJava モデルスコープ適用後に Stage B 詳細へ昇格。フィールド型・継承・実装関係を走査class ノード + 可視性記号。継承 <|-- / 実装 <|.. / 使用 -->
PackageJava モデルパッケージ名でクラスを集約し、継承 / 実装 / フィールド型の参照をパッケージ間に集計package ノード（クラス数付き）+ パッケージ間 -->
SequenceJava モデル（メソッド本体）起点 Class.method から呼び出し木を DFS 展開（深さ制限・循環検出）participant + メッセージ矢印。if→opt/alt、loop→loop、switch→alt、try→group、synchronized→critical
ActivityJava モデル（1 メソッド本体）起点メソッド本体の制御構造を順次走査（呼び出し先は展開せずノード化）if/elseif/else、while/repeat、switch/case、try→partition
ComponentAndroid 解析AndroidManifest の 4 種コンポーネント + intent-filter の action を解析component（<<Activity>> 等のステレオ）+ action usecase → -->
DependencyGradle 解析build.gradle(.kts) の dependencies を解析。libs.versions.toml も解決component（module / library / external）+ scope 別の --> / ..>
ManifestAndroid 解析<application> 属性 + 4 種コンポーネント + permission / feature を sourceSet 別に集計Application rectangle を中心に package でグループ化、包含 *--
Layout選択した layout XMLView 階層を再帰分類（ViewGroup / View / include / fragment / merge）入れ子 rectangle。ノード / 深さ上限超過は truncated note
CommonJava モデル全クラスの fan-in（継承 / 実装 / フィールド / 引数 / 戻り値での被参照数）を集計し上位 N（既定 20）を抽出ハブクラスを <<common>> 強調色、参照元から ..> : uses
Navigation選択した navigation XMLDestination（fragment / activity / dialog 等）と Action（id 参照）を解析state ノード + -->、開始は [*] -->、Deep Link は note
ModuleJava モデル（module-info）module-info.java の requires / exports / opens / uses / provides を抽出component。requires→-->、requires transitive→==>、exports / opens は note
InheritJava モデルClass 図と同じ生成器を、関係線のみ（フィールド / メソッド非表示・上→下レイアウト）に固定継承 <|-- / 実装 <|.. のツリー。兄弟 5 個超で折返し
Call GraphJava モデル（メソッド本体）起点から呼び出し先を DFS 再帰展開（既定 maxDepth=4）。循環はマークWBS（* / ** / ***）。プロジェクト内は色付き、循環は [↩]
Screen Flowプロジェクト再走査Java / Kotlin を正規表現で走査し、Intent 起動・Screen.push・Fragment 遷移を検出state + 遷移種別ごとに色分けした矢印。重複は ×N
Soongプロジェクト再走査配下の Android.bp を軽量パースし、module 名と *_libs / required 等の依存キーを抽出component（cc / java / android 等カテゴリ別 package）+ 依存 -->。外部は external 枠
もっと詳しく: 各生成器のファイル位置・固有ロジック（fan-in 集計・BFS 深さ・truncate 上限など）は 開発者リファレンスの図種カタログに整理しています。

図種切替時の連動（ツールバー ⇄ ツリー ⇄ タブ）

図種を切り替えると、アクティブな題材のまま別図種のタブが開く/フォーカスされ、その結果がツリーのハイライトとツールバー/メニューの選択へ逆方向に同期されます。

各図種の詳細操作

クラス図 (Class)

クラス・インターフェース・列挙体の構造と相互関係を表示。プリセットで密度を切替できます。

プリセットSC内容
MinimalCtrl+1フィールド非表示・関係線のみ
BalancedCtrl+2デフォルト（フィールド + 主要メソッド）
DetailedCtrl+3フィールド + 全メソッド + アノテーション

プレビュー上のクラス名クリックで 1 ホップ近傍の詳細図へドリルダウン、メソッド名横の ▶ で Sequence / Activity を選ぶポップアップ。

シーケンス図 (Sequence)

呼び出し連鎖を時系列で図示。制御構造は if→opt/alt、loop、switch→alt、try→group、synchronized→critical に変換されます。起点は左ツリーのメソッド選択、または Diagram → Choose Sequence Entry…。Filter Sequence Participants… で participant を絞り込めます。

アクティビティ図 (Activity)

1 メソッド本体の制御フロー（if / while / switch / try）をフローチャート化。起点はメソッド選択または Choose Activity Method…。

コールグラフ図 (Call Graph)

起点メソッドから呼び出し先を WBS 形式で可視化。プロジェクト内は黄色、起点は水色で強調。循環は [↩]。既定の再帰深さは 4（Style Settings で変更可）。

共通クラス図 (Common)

fan-in（被参照数）上位 N 件（既定 20）を強調表示。設計ボトルネック候補の発見に。

その他の図種

  • Component / Manifest / Dependency: AndroidManifest / Gradle 解析から自動生成。libs.versions.toml も解決。
  • Layout / Navigation: Choose Layout File… / Choose Navigation Graph… でファイルを選択。
  • Module: module-info.java の JPMS 関係（requires/exports/opens）。
  • Inheritance: extends / implements 階層のみの軽量図。
  • Screen Flow: Intent 起動 + Car App Library Screen.push を統合した状態遷移図。
  • Soong: プロジェクト下の Android.bp を走査し依存をコンポーネント図化。

ドリルダウンとタブナビゲーション

プレビュー上でクラス名にカーソルを置くと指差しアイコンに変わります。クラス名左クリックで DETAILED 詳細図タブ、メソッド横 ▶ で Sequence / Activity ポップアップ。

Juml は VS Code 風のタブ中心モデルです。同じ題材・図種のタブは重複生成されず既存タブにフォーカスが移ります。図の行き来はタブで行います（Back / Forward は現行版にありません）。

タブのライフサイクル

1 つの図タブ（題材 + 図種のキー）は、生成 → 非同期描画 → 表示 → フォーカス/非フォーカス → 破棄、という状態をたどります。ドリルダウンは別キーの新しいタブを生成します。

操作の状態管理

Juml は「タブ＝エディタ」モデルです。状態は タブごとに保持される状態 と、アクティブタブを映すグローバル状態（DiagramState）の二層に分かれます。スコープ／フィルタ等のダイアログやエクスポートは、常に「いまフォーカスしているタブ」の状態を見て動きます。

タブごとに保持される状態

各図タブは自分の描画リクエスト（DiagramRequest）と描画結果を独立して持ちます。別タブへ切り替えても、それぞれの図種・スコープ・起点・ズーム率は保たれます。

状態項目内容変わる操作保持・復元
図種 kindそのタブが描く図（Class / Sequence / …）図種切替・ドリルダウンタブごと。閉じると破棄
スコープ scopeクラス図の表示範囲（パッケージ / 関連線 / 可視性）Diagram › Scopeタブごと。変更で再描画
起点 entryシーケンス / アクティビティ / コールグラフの Class.methodメソッド選択 / Choose … Entryタブごと
非表示 participantシーケンス図で隠す相手Participant フィルタタブごと。変更で再描画
ズーム率プレビューの倍率ズーム・パンタブごとに独立（共有しない）
PlantUML / SVG最後に描画したソースと図再描画時タブごと。ソースタブ／エクスポートが参照

アクティブタブとグローバル状態の同期

タブをフォーカスすると、そのタブの状態が「現在の状態」（DiagramState）へ反映され（mirrorToState）、ツリーのハイライト・ツールバーの図種・ステータスバーが連動します。逆にスタイルやスコープを変えるとアクティブタブが再描画され、結果がまた DiagramState に書き戻されます。

無限ループ防止: タブ → ツリーのハイライトは suppressNotify 付きで行われ、ツリー側の選択コールバックを再発火させません（タブ ⇄ ツリー ⇄ ツールバーの循環を断ち切ります）。
内部構造（クラスの責務）
  • DiagramState = グローバル状態の保持のみ（副作用なし）。フィールド: sequenceEntry / activityEntry / callGraphEntry / currentLayoutKey / currentNavigationKey / currentScope / currentPuml / currentSvgXml / sequenceHiddenParticipants。
  • DiagramController = 状態遷移と UI 同期。onTabFocused → syncToFocusedTab()（ツリー選択）＋ reflectKindInToolbar()、スコープ等は applyStateToActiveTab()。
  • DiagramTabPane / DiagramTab = タブのライフサイクルと描画。startRender()（SwingWorker）、mirrorToState()、setActiveTabSpecAndRender()、rerenderAllTabs()。
  • mirrorToState() は (1) タブ選択時（handleTabSelectionChanged）と (2) 描画完了時（アクティブタブのとき）に呼ばれます。

状態の生存期間

  • タブを閉じると、そのタブの状態は破棄されます。
  • プロジェクトを開き直すと、起点・スコープ・非表示 participant などアクティブタブ状態はクリアされ、既定の Package 図が 1 枚開きます。
  • 一方、スタイル・プリセット等のプロジェクト設定は永続化され、次回ロード時に自動復元されます（→ 設定の自動保存）。

Impact 解析

あるクラスやメソッドに依存している箇所（逆参照）を BFS で辿り、変更影響範囲を把握します。

  1. 右ペインの Impact タブをクリック
  2. Symbol に完全修飾名を入力（クラス com.example.MyClass / メソッド com.example.MyClass.myMethod）
  3. Depth スピナーで BFS 深さを指定（既定 3、範囲 1–10）
  4. Analyze で層別の呼び出し元ツリーを表示
ヒント: 初回は参照インデックス構築に時間がかかります。2 回目以降はキャッシュが利用されます。

巨大な図が描画できないとき（Graphviz）

クラス図などが大きすぎて GUI プレビューで 「Couldn't render this diagram」 になる場合、純 Java の Smetana レイアウトエンジンが破綻しています。より堅牢な Graphviz (dot) を使うと描画できます。

方法手順
GUI から有効化（再起動不要）メニューの Settings → Enable Graphviz (dot)… を選択。PATH / 環境変数 / 同梱バイナリを再検出し、見つからなければ dot 実行ファイルの場所を指定できます。有効化すると開いている図がその場で再描画されます。
環境変数で指定起動前に GRAPHVIZ_DOT=/path/to/dot（または -Dnet.sourceforge.plantuml.GRAPHVIZ_DOT=…）を設定すると、起動時に自動検出されます。
配布物へ同梱bundle/graphviz/<platform>/dot（例 linux-amd64 / mac-aarch64 / windows-amd64）に dot を置くと jar 隣接バイナリとして自動検出。Linux/Mac は ./gradlew copySystemDot で PATH 上の dot をコピーし、./gradlew makeZip で配布 ZIP に同梱できます（動的リンクのため同一 OS / ライブラリ環境でのみ動作）。
PNG エクスポートで切れる場合: 既定でキャンバス上限を 16384 へ引き上げ、scale max で自動縮小しています。それでも収まらない極端な図は、ベクターで上限の影響を受けない SVG エクスポートを使うか、Diagram → Preset / Scope… で範囲を絞ってください。
関連: Smetana / キャンバス上限の詳細は 開発者リファレンスの既知の制約、dot 検出順は PlantUML 描画を参照。
