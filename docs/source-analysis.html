ソース解析パイプライン

core/formats がソースコードから何を取得し、どのように PlantUML へ変換して UML 図を生成するか。Java 解析・データモデル・各図種の生成ロジックをソースから棚卸し。

対象: core/formats サブパッケージ java / uml / android 図種 15
このページは src/main/java/juml/core/formats/ のソースを直接棚卸しした解析エンジンの内部解説です。 CLI/GUI からの使い方は 開発者リファレンス / GUI マニュアル を参照してください。

解析パイプライン全体

入力（ソースファイル / プロジェクトルート）から UML 図までの流れは、大きく スキャン → パース → データモデル構築 → 図生成 の 4 段です。エンジンには 2 つの入口があります。次のシーケンス図は、1 つの .java ソース文字列を JavaStructureExtractor.extract() に渡してから PlantUmlClassDiagram に至るまでの内部呼び出しを表します。

① Java / AIDL / Kotlin の構造抽出

ファサードは UmlGenerator。拡張子で適切なパーサへ振り分け、パース後にステレオタイプを付与します。

段担当処理
振り分けUmlGenerator.extractFromSource.aidl→AidlParser / .kt(s)→KotlinLightScanner / それ以外→JavaStructureExtractor
パースJavaStructureExtractor → JavaParserFrontendJavaParser + JavaSymbolSolver で AST を JavaClassInfo 群へ変換
装飾AaosPattern / JetpackPattern各クラスへ AAOS / Jetpack ステレオタイプを付与

② プロジェクト全体の解析

入口は UmlGenerator.extractFromProjectDetailed()。AOSP 級の巨大ツリーでも回るよう、並列パースと遅延詳細化を組み合わせます。

  1. AndroidProjectScanner.scan(root, opts) で対象ファイル（.java/.aidl/.kt）を収集
  2. JpSolver.fromSourceRoots() でシンボル解決器を構築（FULL モード時のみ）
  3. ExecutorCompletionService（ワーカ数 = CPU − 1）で ParseTask を並列実行
  4. 各ファイルの結果を ClassIndex に put()
  5. mergeManifestInto() で AndroidManifest.xml を解析し各クラスへ androidComponentType を設定
  6. DependencyJarIndex.build() で依存 JAR/AAR の遅延ロード用カタログを構築
  7. ProjectParseResult（classes + ClassIndex + DependencyJarIndex）を返す

③ Android プロジェクト解析

Gradle / Manifest / layout / navigation を束ねる入口が AndroidProjectAnalyzer.analyze()。libs.versions.toml を先読みし、settings.gradle でサブプロジェクト構造を確定してから各 build.gradle / XML を解析し、AndroidProjectAnalysis に集約します。

Stage A / Stage B（2 段階解析）

巨大プロジェクトでメモリと時間を抑えるため、まずヘッダだけを読み（Stage A）、図に必要なクラスだけ本体を再パース（Stage B）します。

段階取得内容メソッド
Stage A（ヘッダのみ）package / simpleName / kind / modifiers / superClass / interfaces / annotations / moduleDirectivesJavaStructureExtractor.extractHeadersOnly()
Stage B（詳細）Stage A + fields / methods / enumConstants / comment / 本体の statement treeJavaStructureExtractor.extract()

Stage B への昇格は ClassIndex.detail(qn, listener) がオンデマンドで実行し、detailedCache（ConcurrentHashMap）で再パースを回避します。JavaClassInfo.isDetailed() がその状態を表します。

プロジェクト全体の解析は GUI からは非同期（SwingWorker）で実行されます。次のシーケンス図は ProjectLoader.start() がバックグラウンドで解析し、完了後に EDT でツリー更新・既定タブ表示まで行う流れです。

Statement ツリー（メソッド本体の構造化）

メソッド本体は、平坦なテキストではなく構造化文ツリーとして保持されます。これがシーケンス図・アクティビティ図・コールグラフ・逆参照の共通基盤です。

Statement サブタイプ: CallBlockReturnThrowYieldBreakContinueLocalVarInlineComment

Call が保持する追加情報:

フィールド内容
receiverレシーバ文字列（空=this コール）
methodName呼び出しメソッド名
resolvedOwnerFqnシンボル解決済み宣言型 FQN（JpSolver が設定）
resolvedSignature解決済みシグネチャ
firstArgLabel第 1 引数が定数シンボルの場合のラベル
inlineMethodsラムダ / 匿名クラス本体から抽出したコールバックメソッド

Block の Kind は 7 種（IF / WHILE / FOR / DO_WHILE / SWITCH / TRY / SYNCHRONIZED）で、各 Block は Branch（条件ラベル付きの分岐）を子に持ちます。この共通表現が、後段でシーケンス／アクティビティの異なる PlantUML 構文へ射影されます。

字句解析（JavaLexer）

JavaParser を使う前段や軽量処理向けに、自前の JavaLexer / JavaToken を備えます。

  • 行コメント（//）・ブロックコメント（/* */）はトークン列に含めずスキップ
  • Java 15+ テキストブロック（""" … """）も STRING トークンとして処理
  • 演算子は 3 文字（>>> <<= ... 等）→ 2 文字（== -> :: 等）→ 1 文字の順で貪欲マッチ
  • ( ) { } [ ] , ; . : @ ? は PUNCT、その他記号は OP に分類
  • expandUnicodeEscapes() が JLS §3.3 の \uXXXX を展開（偶数個バックスラッシュは展開しない）

JavaToken は type / text / line（1-based）/ start / end（半開区間） を保持します。

JavaParser フロントエンド

本体の構造抽出は JavaParserFrontend が JavaParser ライブラリへ委譲します。

  • LanguageLevel.BLEEDING_EDGE で最新言語仕様に寛容
  • パース前に JavaLexer.expandUnicodeEscapes() を適用
  • JpSolver が渡されれば JavaSymbolSolver を仕込んだ JavaParser を使用
  • 構文エラーがあっても isSuccessful()==false の部分結果を継続使用
  • 完全失敗時は正規表現で型宣言キーワードを拾う recoverSkeletons() にフォールバック

Adapter 群の責務分担

AST → データモデルの変換は、ノード種別ごとに専用 Adapter が担います。

Adapter担当
TypeDeclAdapterclass / interface / enum / @interface / record を JavaClassInfo へ。ネスト型は enclosingClass を設定してフラットに追加。extends→superClass、implements/permits→interfaces
MemberAdapterフィールド（int a, b; は変数ごとに分割）・メソッド・コンストラクタを生成。インタフェースの非 default/static メソッドは isAbstract。headersOnly 時は本体パースをスキップ
StatementAdapterメソッド本体を Statement ツリーへ。制御構造を Block+Branch に構造化（次節以降のマッピング元）
ExpressionAdapter式から Call を抽出。レシーバ文字列の構築、第 1 引数のラムダ／定数の扱い、Foo::bar メソッド参照の Call 化

条件式内の Call 持ち上げ: if/while/for/synchronized の条件式中のメソッド呼び出しは、ExpressionAdapter.emitCalls() で Block の直前の兄弟 Call として配置されます（条件評価の副作用を図に残すため）。

シンボル解決（JpSolver）

呼び出し先の「実際の宣言型」を解決し、Call.resolvedOwnerFqn を埋めるのが JpSolver の役割です。シーケンス図の participant 同定の精度を左右します。

  • CombinedTypeSolver に ReflectionTypeSolver(false) + JavaParserTypeSolver(sourceRoot) を登録
  • ソースルートは標準レイアウト src/<set>/java|kotlin 配下から正規表現で自動推定
  • JavaParser はスレッド非安全のため、ワーカ毎に newParser() で生成
  • 解決失敗（UnsolvedSymbolException 等）は無視し、resolvedOwnerFqn=null のまま続行（best-effort）

ステレオタイプ判定

クラスの「役割」を推定し、図にステレオタイプ（<<CarManager>> 等）として反映します。

判定器ロジック（抜粋）
AaosPatternAIDL_INTERFACE→"AIDL" / Car…Manager+AAOS pkg→"CarManager" / Car…Service→"CarService" / ICar… interface→"ICarInterface" / API アノテーション→"AaosApi"。さらに @hide/@SystemApi/@TestApi の可視性、.Stub 継承の binder、@AddedIn 等の API レベルバッジ
JetpackPatternsuperClass 単純名で Fragment / ViewModel 系を判定。アノテーション短名で @AndroidEntryPoint / @HiltViewModel / @HiltAndroidApp / @Module+@InstallIn を、コンストラクタ @Inject で Injectable を判定
AndroidSuperclassDetectorClassIndex の superClass を最大 16 段辿り、Application / Activity / Service / Receiver / Provider / Fragment の基底クラス（FQN・単純名両対応）を特定。Manifest に現れない Fragment の検出に有用

外部 JAR からのヘッダ抽出

依存ライブラリのクラスは ExternalClassReader が ASM で読み取ります。ClassReader を SKIP_CODE | SKIP_DEBUG | SKIP_FRAMES で走らせ、宣言・親クラス・実装インタフェース・公開メソッドシグネチャ・公開フィールドのみを取得し、origin = EXTERNAL_JAR の JavaClassInfo を返します。これにより、シーケンス図などで外部クラスを <<external>> として正しく扱えます。

クラス図 / パッケージ図の生成

PlantUmlClassDiagram

入力は List<JavaClassInfo> + Options。次の順で PlantUML を組み立てます。

  1. Kind.MODULE を除外、excludeExternalLibraries / publicOnly でフィルタ、maxClasses で切り詰め
  2. 全クラスに C{seq} エイリアスを発行（a.b.c の名前空間誤解釈を防止）
  3. groupByPackage なら package "…" { } に分類して emitClass()
  4. PlantUmlClassRelations.emitInheritance() → 継承 <|-- / 実装 <|..
  5. emitUsage() → フィールド型から利用矢印 ..>
  6. PlantUmlClassLegend.emit() で凡例

emitClass() が出力する内容: キーワード（class/abstract/interface/enum/annotation）+ ステレオタイプ（Origin / AAOS / API 可視性 / binder / API バッジ / Car App Library / androidComponentType / aidl / record / sealed / Jetpack）+ コメント（INLINE は .. text ..、NOTE は note top of）+ フィールド（{vis}[{static}][{final}][@Ann] name: Type）+ メソッド（{vis}[…][@Ann] name(p: Type): Ret）。interactiveLinks 時は [[juml://class/FQN]] 等のクリックリンクを付与。

利用関係のターゲットは PlantUmlClassRelations.pickUsageTarget() が、Map<String, Foo> のようなフィールド型からジェネリクスを深さ 1 で展開し、既知クラス名と照合して決定します。

PlantUmlPackageDiagram

各パッケージを package "name\nN classes" として出力し、extends / implements / フィールド型（pickUsageTarget で 1 段）を集計。異なるパッケージ間のみ P1 --> P2 を引き、既定では自パッケージ内参照（self-loop）を抑制します。

シーケンス図の生成

入力は List<JavaClassInfo> + 起点 Class.method + Options。起点メソッドの Statement ツリーを再帰走査して呼び出し連鎖を描きます。

  1. 起点を Caller -> EntryClass: method() として出力
  2. walkStatements() で Statement ツリーを再帰走査
  3. emitCall() が呼び先を解決: resolvedOwnerFqn → receiver 文字列 → フィールド型 → receiver 先頭シンボル の順
  4. SeqEmitters.emitBlock() が制御構造を PlantUML へ変換（下表）
  5. return/throw は note over Participant として表示
  6. 走査後 resolveParticipantOrigins() で外部 JAR を解決し <<external>>/<<missing>> を付与

制御構造マッピング（SeqEmitters）

Java 構文PlantUML
if（分岐なし）opt 条件 … end
if / else if / elsealt 条件 / else 条件2 / end
while / for / do-whileloop while(cond) / for(header) / do-while(cond) … end
switchalt switch(sel)/case X / else case Y / end
try / catch / finallygroup try / else catch E / else finally / end
synchronizedcritical synchronized(lock) / end

participant の色分け

project（自プロジェクト, 既定 #LightSkyBlue）<<external>>（EXTERNAL_JAR）<<missing>>（JAR 不在）<<inline>>（ラムダ/匿名コールバック）<<binder>>（AIDL Stub 実装）

コメントは AT_CALL_SITE（既定: 各呼び出し直下に note right of）と PARTICIPANT_TOP（participant 宣言直後に集約）の 2 モード。forEach 系 + コールバックは loop で囲みます。

アクティビティ図の生成

シーケンス図と同じ Statement ツリーを使いますが、対象は1 メソッド内部のフロー（クラス間展開なし）で、PlantUML の activity 構文へ射影します。

Java 構文PlantUML（activity）
if / else if / elseif (cond) then (yes) / elseif / else (no) / endif
whilewhile (cond) is (true) … endwhile (false)
for / for-eachwhile (for: header) is (loop) … endwhile (done)
do-whilerepeat … repeat while (cond) is (yes) not (no)
switchswitch (expr) / case (X) … endswitch
try / catch / finallypartition "try" / "catch (E)" / "finally" { }
synchronizedpartition "synchronized(lock)" { }

Call → :receiver.method();、LocalVar → :Type var = expr;、InlineComment → note right、ラムダ/コールバックは partition "method → callback()" { } で展開。開始/終了/例外終端は start/stop/kill。

コールグラフ図の生成

PlantUmlCallGraphDiagram は WBS（@startwbs … @endwbs）でメソッド呼び出し階層を描きます。DFS で callDepth×2=クラスノード、callDepth×2+1=メソッドノードの深さを与え、パス上の祖先に同じ className.methodName があれば [↩] を付けて展開を停止（循環検出）。プロジェクト内クラスは色付け、外部クラスは色なしです。

共通クラス図（fan-in 集計）

PlantUmlCommonClassesDiagram.analyze() が「他クラスから多く参照されるハブ」を抽出します。

  1. 外部ライブラリを除いたクラス集合を構築
  2. 各クラスの参照対象型を collectReferencedTypes() で収集: superClassinterfacesフィールド型（ジェネリクス込み）メソッド戻り値型メソッド引数型
  3. resolveKnown() で型文字列を既知クラス QN に解決（FQN→単純名フォールバック）
  4. 参照元 QN を referrersByQn に蓄積
  5. 参照回数の降順（同数は QN 昇順）でソートし上位 N（既定 20）を返す

図ではハブを #LightGoldenRodYellow + <<common>> + N refs で強調し、参照元から referer ..> hub : uses の破線矢印を引きます。設計上のボトルネック候補の発見に使えます。

モジュール図の生成

PlantUmlModuleDiagram は Kind.MODULE（module-info.java）のみを対象に component "moduleName" ノードを出力。requires transitive → ==>、通常 requires → -->、exports/opens は note bottom of に列挙します。
